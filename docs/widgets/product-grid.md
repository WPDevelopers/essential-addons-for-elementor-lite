# Product Grid Widget

> Renders a WooCommerce product grid with grid / masonry / list layouts, twelve preset templates loaded from per-file PHP templates, AJAX-paginated load-more + infinity scroll, Quick View popup, native product comparison (lite-built into this widget), and WPC Wholesale / YITH Wishlist passthroughs. The PHP class is exceptionally large (4063 lines) — most of the bulk is style controls per preset. Functionally distinct from `Woo_Product_List` despite shared `list` layout templates.

**Class file:** [`includes/Elements/Product_Grid.php`](../../includes/Elements/Product_Grid.php)
**Slug:** `product-grid` (widget id `eicon-woocommerce`) ⚠️ — `get_name()` returns `eicon-woocommerce` while [`config.php`](../../config.php#L569) registers under slug `product-grid`. Asset_Builder loads assets keyed by `product-grid`; the `elementor/frontend/element_ready/eicon-woocommerce.default` JS hook uses the widget name. See [`docs/architecture/asset-loading.md`](../architecture/asset-loading.md) for the slug-vs-widget-id resolution.
**Public docs:** <https://essential-addons.com/elementor/docs/woocommerce-product-grid/>
**Pro-shared:** ✅ Yes — Pro extends the same `eicon-woocommerce` widget name through two mechanisms: (1) drops additional preset PHP files into `EAEL_PRO_PLUGIN_PATH/includes/Template/Eicon-Woocommerce/`, which `Template_Query::get_template_files()` auto-discovers, (2) emits Pro-only controls (e.g. `eael_product_rating_type`) gated behind `apply_filters('eael/pro_enabled', false)` checks inline in [`register_controls()`](../../includes/Elements/Product_Grid.php#L761).

## Overview

Product Grid is the canonical EA WooCommerce widget — a WP_Query-driven product loop where each rendered product is generated by `include`-ing the preset's PHP template inside the loop ([line 3763](../../includes/Elements/Product_Grid.php#L3763)). Twelve free presets ship in [`includes/Template/Eicon-Woocommerce/`](../../includes/Template/Eicon-Woocommerce/) (simple, reveal, overlay, default, preset-1 through preset-8, plus four list-preset variants); Pro extends by adding more files into its parallel directory.

Beyond the post listing, the widget integrates **four discrete subsystems**: Load More (shared AJAX pagination from Post_Grid), the native EA Product Comparison popup (this widget owns `eael-wcpc-modal` and its AJAX endpoint), Quick View (loaded via the `quick-view.min.js` shared asset), and YITH WCWL Wishlist passthrough. The render method swaps preset slug → template file → include — adding a new preset means dropping a `.php` template with a `Template Name:` header into the template dir, no PHP class changes needed.

## Architecture

- **Template-include rendering** — `render()` calls `get_template($preset_slug)` (which strips `eael-product-` prefix at [line 3722](../../includes/Elements/Product_Grid.php#L3722)) and `include`s the matching template inside the WP_Query loop ([line 3763](../../includes/Elements/Product_Grid.php#L3763)). Adding a new preset = drop a PHP file with `/** Template Name: Foo */` header in `includes/Template/Eicon-Woocommerce/`. No PHP class changes.
- **Three-tier template directory merge** — `Template_Query::get_template_files()` scans Lite dir + Pro dir + theme override dir (`wp-content/themes/<active-theme>/Template/Eicon-Woocommerce/`) and merges them. Themes can override presets without touching the plugin. Pro adds presets without forking Lite. ⚠️ Naming collision: if Lite + Pro + theme all ship a `simple.php`, the **last-scanned** wins (theme > pro > lite — but ordering depends on `scandir` sort flag `1` which is alphabetical).
- **Widget name ≠ slug** — `get_name()` returns `eicon-woocommerce` (legacy from when the widget icon was reused as the registration handle). `config.php` registers under slug `product-grid`. Asset_Builder reads `config.php` so assets load correctly when the slug matches the element key; the `elementor/frontend/element_ready/eicon-woocommerce.default` JS hook keys off the widget name. The mismatch surfaces in: template directory derivation (`Eicon-Woocommerce`), JS init hook, and the legacy `eicon-woocommerce` term inside `apply_filters( 'eael_product_wrapper_class', [], $product_id, 'eicon-woocommerce' )`.
- **No `eael_section_pro` upsell** — instead, Pro features are emitted inline behind `apply_filters('eael/pro_enabled', false)` checks ([line 761](../../includes/Elements/Product_Grid.php#L761)). Anywhere in the codebase setting that filter to `true` flips this widget's panels to expose Pro controls. Mechanism differs from the standard upsell pattern in [`_patterns.md § eael_section_pro`](_patterns.md#eael_section_pro-standard-upsell-panel).
- **Two render paths for the same widget** — (1) initial server-side render in `render()` uses the template-include pattern; (2) AJAX load-more uses `Ajax_Handler::ajax_load_more()` which also includes the same template files, but the path passes through `render_template_()` in [`Template/Content/Product_Grid.php`](../../includes/Template/Content/Product_Grid.php) for older alternate-render flows. The two paths must stay consistent or the load-more output mismatches initial render.
- **Product Compare modal lives inside Product_Grid** — `Woo_Product_Comparable` trait, modal markup injected into `<body>` by [`product-grid.js`](../../src/js/view/product-grid.js#L22), AJAX endpoint `wp_ajax_eael_product_grid` builds the comparison-table HTML, state persists in `localStorage.productIds`. Other WooCommerce widgets can use compare by including the same trait.
- **Secondary image hover swap** — when `eael_product_grid_show_secondary_image` is `yes`, templates emit `data-src` and `data-src-hover` on `.eael-product-wrap`; JS handler at [lines 285-316 of product-grid.js](../../src/js/view/product-grid.js#L285) swaps `<img src>` and `srcset` on mouseover. Responsive: 3 separate switchers per device (desktop/tablet/mobile) write `data-ssi-{device}` and JS reads `body[data-elementor-device-mode]` to pick.
- **WPC Wholesale / Whols Lite passthrough** ([line 4020](../../includes/Elements/Product_Grid.php#L4020)) — if `whols_lite()` exists, applies the `woocommerce_product_query_meta_query` filter to the widget's query args so wholesale price visibility rules carry over. No equivalent generic plugin-interop pattern documented; this is one-off.

For shared patterns (Liquid Glass / FA4 / WPML / `has_pro` / `eael_section_pro`), see [`_patterns.md`](_patterns.md). Product_Grid uses **none** of those — it has its own bespoke patterns (template directory merge, inline `eael/pro_enabled` gating, dual AJAX render paths).

## JavaScript Lifecycle

- **Trigger:** `elementorFrontend.hooks.addAction('frontend/element_ready/eicon-woocommerce.default', productGrid)` ([line 324-327 of product-grid.js](../../src/js/view/product-grid.js#L324)) — keyed by widget name `eicon-woocommerce`, not slug `product-grid`
- **Outer guard:** `if (eael.elementStatusCheck('eaelProductGridLoad') && window.forceFullyRun === undefined) return;` — prevents re-init in Elementor preview
- **Boot pattern:** Uses `eael.hooks.addAction("init", "ea", …)` (newer EA pattern) instead of `jQuery(window).on("elementor/frontend/init", …)` — see [`_patterns.md`](_patterns.md)
- **Vendor deps:** Isotope (for masonry), imagesLoaded (defer layout), jQuery, Elementor's frontend
- **Reads on init:** `#eael-product-grid[data-widget-id|data-page-id|data-nonce]`; per-`.eael-product-wrap`: `data-src`, `data-src-hover`; per-`.products`: `data-layout-mode`, `data-ssi-{device}`
- **Three click delegations bound on `document`:**
  - `.eael-wc-compare` → AJAX `action: eael_product_grid` — pushes product ID to `localStorage.productIds`, opens `.eael-wcpc-modal`
  - `.eael-wc-remove` → AJAX `action: eael_product_grid` with `remove_product=1`
  - `.close-modal` → hide modal + overlay
- **Pagination delegation on `$scope`:** `.eael-woo-pagination a` → 2 parallel AJAX requests (product list + pagination bar), scrolls window to top of grid
- **Inline render-time `<script>` block** ([line 3805-3826](../../includes/Elements/Product_Grid.php#L3805)) — Isotope init duplicated in PHP for editor preview (jQuery DOM-ready); same logic also runs in [product-grid.js line 69-73](../../src/js/view/product-grid.js#L69) (but in JS-side this triggers a window-resize on ajaxComplete rather than calling Isotope directly — inconsistent)
- **Quick View bootstrap:** dispatches `eael.hooks.doAction("quickViewAddMarkup", $scope, $)` and `quickViewPopupViewInit` — wires the shared Quick View plugin if loaded
- **Body popup wrapper appended once:** `.eael-woocommerce-popup-view` is appended to `<body>` if not already present
- **Secondary-image hover swap:** `mouseover/mouseout` delegated to `.eael-product-wrap`, swaps `<img src>` and `srcset` based on `data-ssi-{device}`; reads `body[data-elementor-device-mode]` to know current device — relies on Elementor's responsive preview
- **AJAX in `editorMode`:** if `isEditMode === true`, force-shows `.eael-product-image-wrap .woocommerce-product-gallery` opacity to 1 — Elementor's product gallery wrapper hides by default

## Common Issues

### Masonry layout overlaps after page resize or Load More

- **Likely cause:** Isotope `layout()` ran before images settled; or the `$(window).trigger('resize')` on `ajaxComplete` ([line 70-72 of product-grid.js](../../src/js/view/product-grid.js#L70)) fired before append completed
- **Diagnose:** browser console for Isotope errors; check that `data-layout-mode="masonry"` is on `.products`
- **Fix:** hard refresh; if persistent, verify `imagesloaded.pkgd.min.js` loaded (look in network panel). The inline `<script>` from render() may re-init and conflict with `product-grid.js` — if a regression appears, check which one fires last

### Compare button doesn't open modal / modal stuck open

- **Likely cause:** `data-nonce` mismatch (stale page cache) → AJAX returns 403, modal injection silently fails. Or `.eael-wcpc-modal` was appended twice (multiple Product_Grid instances) and the second close-handler targets the wrong instance
- **Diagnose:** browser DevTools → network → look for `admin-ajax.php` POST with `action=eael_product_grid`; check response body. Inspect `<body>` — should have exactly one `.eael-wcpc-modal`
- **Fix:** clear page cache (nonce expires after 24h); for double-modal, the JS at [line 22 of product-grid.js](../../src/js/view/product-grid.js#L22) appends unconditionally — a page with multiple Product_Grid widgets gets multiple modals (known bug, see Known Limitations)

### Pro presets / theme-override presets don't appear in picker

- **Likely cause:** `Template_Query::get_template_files()` scans directories named by `process_directory_name()` → `Eicon-Woocommerce`. If Pro/theme uses any other casing or location, files are skipped. Also templates without a `Template Name:` PHP doc header are filtered out by `get_file_data($path, $template_headers)`
- **Diagnose:** verify the directory exists at exactly `wp-content/plugins/<plugin>/includes/Template/Eicon-Woocommerce/` or `wp-content/themes/<theme>/Template/Eicon-Woocommerce/`; verify each `.php` file starts with `/** Template Name: Something */`
- **Fix:** rename directory to exact case `Eicon-Woocommerce`; add the `Template Name:` header

### Secondary image hover doesn't swap on tablet / mobile preview

- **Likely cause:** `data-ssi-tablet` / `data-ssi-mobile` not emitted because the corresponding switcher is `empty` (default). Or the JS reads `body[data-elementor-device-mode]` and that attr only exists when Elementor's responsive preview is active — on real device viewports the attr defaults to `desktop`
- **Diagnose:** inspect `.products` for `data-ssi-{device}` attrs; check `body[data-elementor-device-mode]` value in DevTools
- **Fix:** enable the per-device switcher in widget settings; for production behaviour beyond Elementor preview, this requires viewport-detect JS (not currently implemented)

### Load More button shows but clicking does nothing

- **Likely cause:** `load-more.js` not loaded (config drift), or `data-class="Essential_Addons_Elementor\Elements\Product_Grid"` doesn't resolve in `ajax_load_more` dispatcher — the class must `class_exists` server-side
- **Diagnose:** browser console — look for "load-more is not defined" or 500 on `admin-ajax.php?action=load_more`; verify `localize.ajaxurl` is defined
- **Fix:** ensure `assets/front-end/js/view/load-more.min.js` is enqueued (Asset_Builder picks it up from `config.php` line 600); if missing, run `npm run build`

### "WooCommerce is not installed/activated" notice in editor

- **Likely cause:** WooCommerce was deactivated; `register_controls()` short-circuits at [line 173-175](../../includes/Elements/Product_Grid.php#L173) (`if (!function_exists('WC')) { return; }`) so only the layout section + warning notice show
- **Diagnose:** check WP admin → Plugins for WooCommerce status
- **Fix:** activate WooCommerce

### Pagination clicks scroll oddly / break on multi-grid pages

- **Likely cause:** `.eael-woo-pagination` click handler [animates `html, body` to `$(widgetclass + " .eael-product-grid").offset().top - 50`](../../src/js/view/product-grid.js#L224) — but `widgetclass` only matches the **first** matching `.elementor-element-{widgetid}` in DOM, so multiple Product_Grid widgets on the same page can race
- **Diagnose:** check that each grid has a distinct `data-widget-id` on `#eael-product-grid`; the `#` selector means `id="eael-product-grid"` is **non-unique** when multiple grids render — a documented bug (see Known Limitations)
- **Fix:** known issue with multiple instances; the `id="eael-product-grid"` should be widget-id-scoped

## Known Limitations

- **`id="eael-product-grid"` is hardcoded** ([line 3697](../../includes/Elements/Product_Grid.php#L3697)) — multiple Product_Grid widgets on one page produce duplicate IDs (HTML validation error) and the JS `$scope.find("#eael-product-grid")` only returns the first match. Workaround in JS scopes via `.elementor-element-{widgetid}` but the inner `#eael-product-grid` selector still resolves to the document-first match.
- **Widget name `eicon-woocommerce` ≠ slug `product-grid`** — historical legacy. Anywhere code keys by widget name (JS hook, template directory, `eael_product_wrapper_class` filter param) doesn't match anywhere code keys by slug (`config.php`, Asset_Builder). Renaming requires migrating both keys plus a backward-compat alias.
- **Dual-signature emissions of `eael_woo_before_product_loop` / `_after_product_loop`** — Product_Grid passes `$preset_slug`; Ajax_Handler passes nothing. Listeners that read the arg crash on the no-arg path. `_after_product_loop` is also fired **twice in the same render** ([line 3768 and 3795](../../includes/Elements/Product_Grid.php#L3768)) — once after the `</ul>` close, once at end of method.
- **`eael_product_grid_products_status` exposes draft / pending / future** ([line 4033](../../includes/Elements/Product_Grid.php#L4033)) — the UI lets editors opt to show non-publish statuses. AJAX handler clamps to `[publish, draft, pending, future]` ([Ajax_Handler line 723](../../includes/Traits/Ajax_Handler.php#L723)) but does **not** check user capability — an unauthenticated `wp_ajax_nopriv_woo_product_pagination_product` request with a tampered `data-args` could request `draft` content. This is a known sensitivity; see [`nopriv-ajax-hardening` skill](../../.claude/skills/nopriv-ajax-hardening) for the audit pattern.
- **`render_template_()` in [`Template/Content/Product_Grid.php`](../../includes/Template/Content/Product_Grid.php) is a parallel render path** — used for some legacy AJAX flows but doesn't share preset templates with `render()`. The inline-only presets (simple, reveal, overlay) are duplicated in this trait's match-arms ([lines 29-113](../../includes/Template/Content/Product_Grid.php#L29)). If a preset template file changes, this trait silently drifts.
- **Inline `<script>` for Isotope** ([line 3805-3826](../../includes/Elements/Product_Grid.php#L3805)) duplicates JS-side Isotope handling — frontend has both PHP-emitted inline init AND `product-grid.js` resize trigger. Race condition possible; if the JS file logic changes, both must update.
- **No `elementStatusCheck` guard around the click delegations** — `productGrid()` returns early via the outer guard, but the click handlers are document-delegated and bound only once per `init` cycle. If Elementor re-fires `frontend/element_ready` (e.g. inside Elementor's preview switching) the delegations get re-bound and each subsequent click fires multiple AJAX requests.
- **`.eael-wcpc-modal` injected into `<body>` unconditionally** ([line 22 of product-grid.js](../../src/js/view/product-grid.js#L22)) — multiple Product_Grid widgets on one page each append their own modal. Only the first is targeted by `getElementsByClassName(...)[0]` at line 31. The duplicates are dead nodes.
- **`add_to_cart_button_custom_text` filter is not removed on early `return`** — if `render()` short-circuits via `if (!function_exists('WC')) { return; }` at [line 3626](../../includes/Elements/Product_Grid.php#L3626), the filter callback was never added either, so this is harmless. But if a future change adds filters before the WC check, leakage is a risk.
- **`eael_product_grid_image_size_size` filter for thumbnails** — `single_product_archive_thumbnail_size` is filtered via `eael_customize_woo_prod_thumbnail_size` to swap sizes; `remove_filter` runs at end of render. If `render()` throws mid-loop, the filter persists and contaminates subsequent widgets on the same page.
- **3069-line SCSS** — preset-specific styles (preset-1 through preset-8) each have ~200 lines of dedicated rules. Cross-preset style drift is a maintenance burden; refactor opportunity to a CSS-variable token system per preset.
- **List-layout share with `Woo_Product_List` widget** — list-preset PHP files visually overlap with `Woo_Product_List` presets but are technically separate (distinct directories, distinct PHP classes). Visual drift between the two is silent until a designer spots it.
- **WPC Wholesale passthrough is hardcoded by name** ([line 4020](../../includes/Elements/Product_Grid.php#L4020)) — `function_exists('whols_lite')` checks the plugin's namespace; if WPC renames their function, this break silently. No equivalent generic third-party-meta_query passthrough hook for other wholesale plugins.
- **No `Pro` upsell panel for Lite users** — unlike most EA widgets, this widget doesn't show `eael_section_pro`. Pro features appear inline behind `eael/pro_enabled` filter gating. Users on Lite see fewer controls but no "Go Pro" indicator within this widget's panels.
